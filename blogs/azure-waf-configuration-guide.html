Azure WAF Configuration: A Comprehensive Guide

Dive deep into Azure Web Application Firewall configuration, custom rules, managed rulesets, and integration with Application Gateway. Learn how to protect your web applications from common threats and vulnerabilities.

Azure WAF Security Application Gateway Web Security

Azure Web Application Firewall (WAF) is a cloud-native service that provides centralized protection for your web applications from common exploits and vulnerabilities. Whether deployed on Azure Application Gateway, Azure Front Door, or Azure CDN, WAF acts as a shield against malicious attacks like SQL injection, cross-site scripting (XSS), and other OWASP Top 10 threats.

Understanding Azure WAF Architecture

Azure WAF operates at the application layer (Layer 7) and integrates seamlessly with several Azure services:

  • Application Gateway WAF: Regional protection for applications
  • Front Door WAF: Global protection at the edge
  • CDN WAF: Content delivery with security

Each deployment model offers unique advantages depending on your application architecture and performance requirements.

WAF Policy Configuration

The foundation of Azure WAF protection lies in its policy configuration. Let's explore how to set up a comprehensive WAF policy:

# WAF Policy Resource
resource "azurerm_web_application_firewall_policy" "main" {
  name                = "wafpol-${var.environment}"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location

  policy_settings {
    enabled                     = true
    mode                       = "Prevention"
    request_body_check         = true
    file_upload_limit_in_mb    = 100
    max_request_body_size_in_kb = 128
  }

  managed_rules {
    exclusion {
      match_variable          = "RequestHeaderNames"
      selector               = "x-company-header"
      selector_match_operator = "Equals"
    }

    managed_rule_set {
      type    = "OWASP"
      version = "3.2"

      rule_group_override {
        rule_group_name = "REQUEST-920-PROTOCOL-ENFORCEMENT"
        
        rule {
          id      = "920300"
          enabled = false
          action  = "Log"
        }
      }
    }

    managed_rule_set {
      type    = "Microsoft_BotManagerRuleSet"
      version = "0.1"
    }
  }
}

Policy Settings Explained

Let's break down the key policy settings:

  • Mode: Detection (logs only) vs Prevention (blocks threats)
  • Request Body Check: Inspects POST request bodies
  • File Upload Limit: Maximum file size for uploads
  • Max Request Body Size: Prevents large request attacks

Best Practice

Start with Detection mode in production to understand your application's traffic patterns before switching to Prevention mode.

Custom Rules Implementation

While managed rules provide excellent baseline protection, custom rules allow you to address specific security requirements:

resource "azurerm_web_application_firewall_policy" "main" {
  # ... previous configuration ...

  custom_rules {
    name      = "RateLimitRule"
    priority  = 1
    rule_type = "RateLimitRule"
    action    = "Block"

    match_conditions {
      match_variables {
        variable_name = "RemoteAddr"
      }

      operator           = "IPMatch"
      negation_condition = false
      match_values       = ["0.0.0.0/0"]
    }

    rate_limit_duration    = "PT1M"
    rate_limit_threshold   = 100
  }

  custom_rules {
    name      = "GeoBlockRule"
    priority  = 2
    rule_type = "MatchRule"
    action    = "Block"

    match_conditions {
      match_variables {
        variable_name = "RemoteAddr"
      }

      operator           = "GeoMatch"
      negation_condition = false
      match_values       = ["CN", "RU", "KP"]  # Block specific countries
    }
  }

  custom_rules {
    name      = "AdminPathProtection"
    priority  = 3
    rule_type = "MatchRule"
    action    = "Block"

    match_conditions {
      match_variables {
        variable_name = "RequestUri"
      }

      operator           = "Contains"
      negation_condition = false
      match_values       = ["/admin", "/wp-admin"]
      transforms         = ["Lowercase"]
    }

    match_conditions {
      match_variables {
        variable_name = "RemoteAddr"
      }

      operator           = "IPMatch"
      negation_condition = true
      match_values       = ["203.0.113.0/24"]  # Allow only from trusted IPs
    }
  }
}

Custom Rule Types

Azure WAF supports several custom rule types:

  1. Rate Limit Rules: Prevent DDoS and brute-force attacks
  2. Match Rules: Block based on specific conditions
  3. Geo-filtering Rules: Control access by geographic location

Integration with Application Gateway

Here's how to integrate WAF with Azure Application Gateway:

# Public IP for Application Gateway
resource "azurerm_public_ip" "gateway" {
  name                = "pip-agw-${var.environment}"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location
  allocation_method   = "Static"
  sku                = "Standard"
}

# Application Gateway with WAF
resource "azurerm_application_gateway" "main" {
  name                = "agw-${var.environment}"
  resource_group_name = azurerm_resource_group.main.name
  location            = azurerm_resource_group.main.location

  sku {
    name     = "WAF_v2"
    tier     = "WAF_v2"
    capacity = 2
  }

  gateway_ip_configuration {
    name      = "gateway-ip-config"
    subnet_id = azurerm_subnet.gateway.id
  }

  frontend_port {
    name = "http-port"
    port = 80
  }

  frontend_port {
    name = "https-port"
    port = 443
  }

  frontend_ip_configuration {
    name                 = "frontend-ip-config"
    public_ip_address_id = azurerm_public_ip.gateway.id
  }

  backend_address_pool {
    name = "backend-pool"
    ip_addresses = var.backend_servers
  }

  backend_http_settings {
    name                  = "backend-http-settings"
    cookie_based_affinity = "Disabled"
    path                  = "/"
    port                  = 80
    protocol              = "Http"
    request_timeout       = 60
  }

  http_listener {
    name                           = "http-listener"
    frontend_ip_configuration_name = "frontend-ip-config"
    frontend_port_name             = "http-port"
    protocol                       = "Http"
  }

  request_routing_rule {
    name                       = "routing-rule"
    rule_type                  = "Basic"
    http_listener_name         = "http-listener"
    backend_address_pool_name  = "backend-pool"
    backend_http_settings_name = "backend-http-settings"
    priority                   = 100
  }

  # WAF Configuration
  firewall_policy_id = azurerm_web_application_firewall_policy.main.id

  # Enable WAF
  waf_configuration {
    enabled          = true
    firewall_mode    = "Prevention"
    rule_set_type    = "OWASP"
    rule_set_version = "3.2"
  }
}

Advanced WAF Features

1. Bot Protection

Azure WAF includes sophisticated bot protection capabilities:

managed_rule_set {
  type    = "Microsoft_BotManagerRuleSet"
  version = "0.1"

  rule_group_override {
    rule_group_name = "BadBots"
    
    rule {
      id      = "200002"
      enabled = true
      action  = "Block"
    }
  }

  rule_group_override {
    rule_group_name = "GoodBots"
    
    rule {
      id      = "200001"
      enabled = true
      action  = "Allow"
    }
  }
}

2. Exclusions and Fine-tuning

WAF exclusions help reduce false positives:

# Global exclusions
exclusion {
  match_variable          = "RequestArgNames"
  selector               = "__VIEWSTATE"
  selector_match_operator = "Equals"
}

exclusion {
  match_variable          = "RequestCookieNames"
  selector               = "session_*"
  selector_match_operator = "StartsWith"
}

# Rule-specific exclusions
managed_rule_set {
  type    = "OWASP"
  version = "3.2"

  rule_group_override {
    rule_group_name = "REQUEST-942-APPLICATION-ATTACK-SQLI"
    
    rule {
      id      = "942100"
      enabled = true
      action  = "Block"
      
      exclusions {
        match_variable          = "RequestArgNames"
        selector               = "searchQuery"
        selector_match_operator = "Equals"
      }
    }
  }
}

Monitoring and Alerting

Effective WAF monitoring requires comprehensive logging and alerting:

# Log Analytics Workspace
resource "azurerm_log_analytics_workspace" "waf" {
  name                = "law-waf-${var.environment}"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  sku                 = "PerGB2018"
  retention_in_days   = 90
}

# Diagnostic Settings for Application Gateway
resource "azurerm_monitor_diagnostic_setting" "gateway" {
  name                       = "gateway-diagnostics"
  target_resource_id         = azurerm_application_gateway.main.id
  log_analytics_workspace_id = azurerm_log_analytics_workspace.waf.id

  enabled_log {
    category = "ApplicationGatewayAccessLog"
  }

  enabled_log {
    category = "ApplicationGatewayPerformanceLog"
  }

  enabled_log {
    category = "ApplicationGatewayFirewallLog"
  }

  metric {
    category = "AllMetrics"
    enabled  = true
  }
}

# Alert Rule for WAF Blocks
resource "azurerm_monitor_metric_alert" "waf_blocks" {
  name                = "waf-high-block-rate"
  resource_group_name = azurerm_resource_group.main.name
  scopes              = [azurerm_application_gateway.main.id]
  description         = "Alert when WAF block rate is high"
  severity            = 2
  frequency           = "PT5M"
  window_size         = "PT15M"

  criteria {
    metric_namespace = "Microsoft.Network/applicationGateways"
    metric_name      = "ApplicationGatewayTotalTime"
    aggregation      = "Count"
    operator         = "GreaterThan"
    threshold        = 100
  }

  action {
    action_group_id = azurerm_monitor_action_group.security.id
  }
}

Performance Optimization

Performance Impact

WAF inspection adds latency to requests. Optimize rules and exclusions to minimize performance impact on legitimate traffic.

Optimization Strategies

  1. Rule Ordering: Place most specific rules first
  2. Exclusions: Use targeted exclusions to reduce false positives
  3. Body Inspection: Disable for endpoints that don't need it
  4. File Size Limits: Set appropriate limits for your use case

Testing and Validation

Regular testing ensures your WAF configuration remains effective:

1. Penetration Testing

# Test SQL injection protection
curl -X POST "https://yourapp.com/login" \
  -d "username=admin' OR '1'='1&password=test"

# Test XSS protection
curl "https://yourapp.com/search?q="

# Test file upload limits
curl -X POST "https://yourapp.com/upload" \
  -F "file=@largefile.zip" \
  -H "Content-Type: multipart/form-data"

2. Automated Testing

  vmImage: 'ubuntu-latest'

steps:

  displayName: 'Test WAF Rules'

Cost Optimization

WAF costs can add up, especially for high-traffic applications. Consider these optimization strategies:

  • Rule Efficiency: Remove unnecessary rules and rule groups
  • Request Sampling: Use sampling for analytics in non-critical environments
  • Log Retention: Adjust log retention periods based on compliance requirements
  • Capacity Planning: Right-size Application Gateway capacity units

Compliance and Governance

For regulated industries, WAF configuration must meet compliance requirements:

# Azure Policy for WAF compliance
resource "azurerm_policy_definition" "waf_required" {
  name         = "waf-required-policy"
  policy_type  = "Custom"
  mode         = "Indexed"
  display_name = "Require WAF on Application Gateways"

  policy_rule = jsonencode({
    if = {
      allOf = [
        {
          field = "type"
          equals = "Microsoft.Network/applicationGateways"
        },
        {
          field = "Microsoft.Network/applicationGateways/sku.tier"
          notEquals = "WAF_v2"
        }
      ]
    }
    then = {
      effect = "deny"
    }
  })
}

Conclusion

Azure WAF provides robust protection for web applications when properly configured and maintained. Key takeaways include:

  • Start with managed rulesets and add custom rules as needed
  • Use Detection mode initially to understand traffic patterns
  • Implement comprehensive monitoring and alerting
  • Regularly test and tune your WAF configuration
  • Consider performance impact when designing rules
  • Maintain compliance through governance policies

Remember that security is an ongoing process. Regularly review WAF logs, update rules based on emerging threats, and conduct security assessments to ensure your applications remain protected against evolving attack vectors.
